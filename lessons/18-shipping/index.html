idea 1Live engine vs baked file — pick your trade-off

Two ways to put this music in a game, and they're opposites:

ship the live engineship a baked file
whatthe synth code + a seed; synthesised every framea pre-rendered audio buffer / .wav, just played back
sizetiny (kilobytes of code)bigger (seconds of audio = ~170 KB/sec stereo)
CPUconstant cost while playingalmost free (just streaming samples)
reactiveyes — change the seed / filter live, react to gameplayno — it's frozen; you only pick which clip

For an acid game the sweet spot is usually the live engine (it reacts, and it's tiny), but you bake when you need zero CPU or exact repeatability. This page does both, from the same code.

idea 2Offline render — faster than real time

The magic tool is OfflineAudioContext: the same Web Audio graph you've been building, but instead of playing to the speakers in real time, it computes the whole thing as fast as the CPU can and hands you an AudioBuffer. Two bars might take a few milliseconds to render. Press ⏺ render above and watch the readout — you'll see the speed-up (often 50–100×).

That buffer is your loop, frozen. Play it back with ▶ play baked (no synth running now — it's just streaming samples), or encode it to a .wav and ⬇ download it. That file is a real, standard audio file — drop it into a game engine, a DAW, anywhere.

// CS corner: new OfflineAudioContext(2, seconds·44100, 44100) → build the identical graph → await ctx.startRendering() → an AudioBuffer. It's the same lookahead scheduling (lesson 10), just against a clock that isn't wall-time. WAV export is a 44-byte header + interleaved 16-bit PCM (float −1..1 → int16), wrapped in a Blob and a download link. No library.

idea 3The mobile gotcha — audio starts locked

The single most common "why is there no sound?" bug. Browsers block audio until the user interacts — an AudioContext is born suspended and stays silent until a real tap or click resume()s it. That's why every lesson has a "power on" button: the first gesture is what unlocks the audio. Watch the badge above flip from 🔒 locked to running the moment you power on.

iOS is strictest: create/resume the context inside the tap handler, and some old versions need a silent buffer played once to fully wake up. Get this wrong and it works on your desktop and is dead silent on every phone — the classic web-audio heartbreak.

// CS corner: audioCtx.state is "suspended" → "running"; a statechange event fires on transition. The unlock must be in a trusted user-gesture handler (pointerdown/click) — you can't resume() from a timer or on load.

idea 4Latency & memory — the last details

  • Latency — the gap between "schedule a note" and "hear it". Live synths schedule a touch ahead (the lookahead window) so gameplay hits feel tight; ctx.outputLatency tells you the hardware delay. Sample playback is effectively instant.
  • Memory — a live engine is a handful of nodes (near nothing). A baked stereo minute is ~10 MB uncompressed; ship a short loop and repeat it, or MP3/OGG-compress it, rather than rendering long stretches.
  • Voices — cap polyphony and reuse nodes; one-shot drum voices should auto-stop (they do here) so they get garbage-collected.

For our arcade game (decided back in the plan: a web game): ship the reusable JS engine and drive it from the game's seed/score, and bake a couple of loops only where you want zero cost. Everything on this page is that engine.
